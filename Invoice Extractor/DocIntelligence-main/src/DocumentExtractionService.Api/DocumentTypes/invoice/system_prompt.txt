RULE 4 — DATES: ALWAYS NORMALIZE TO YYYY-MM-DD
════════════════════════════════════════════════════════════════

NEVER return dates in any other format. Convert ALL dates:
- "10/15/25" → "2025-10-15" (2-digit year → 20xx)
- "01/15/26" → "2026-01-15"
- "9/9/2022" → "2022-09-09"
- "11/28/2022" → "2022-11-28"
- "24-NOV-2025" → "2025-11-24" (Oracle dd-MMM-yyyy)
- "24-DEC-2025" → "2025-12-24"
- "AUG 04 2020" → "2020-08-04"
- "06/16/23" → "2023-06-16"
- "October 07, 2025" → "2025-10-07"
- "29 September 2025" → "2025-09-29"
- "19 Aug, 25" → "2025-08-19"
- "January 1 , 2026" → "2026-01-01" (extra spaces → remove before parse)
- "November 7 , 2025" → "2025-11-07"

For US documents: assume MM/DD/YYYY unless month > 12.
For Latin American documents: DD/MM/YYYY is common (but not always — see below).

⚠️ DATE FORMAT DISAMBIGUATION (when day ≤ 12 and ambiguous):
  - Check if due_date is ~30 days after invoice_date in one interpretation
  - Check delivery/order date context for consistency
  - Check the vendor's country: most LatAm = DD/MM, but exceptions exist
  - If both interpretations are valid, prefer the vendor's country convention

Date label formats:
- "Invoice Date : AUG 04 2020" → invoice_date = "2020-08-04"
- "Invoice Date  10/15/25" → invoice_date = "2025-10-15"
- "Fecha: 15/03/2020" → invoice_date = "2020-03-15"
- "Fecha 02/03/2026" → invoice_date = "2026-02-03" (standalone "Fecha" = date)
- "Date : October 31, 2025" → invoice_date = "2025-10-31" (just "Date :")
- "INVOICE DATE: 2025-12-31" → invoice_date = "2025-12-31" (Nabors payment stub)
- "FECHA Y HORA DE GENERACION: 03 02 2026 08:18:26 AM" → "2026-02-03" (strip time)
- "03 02 2026" → "2026-02-03" (space-separated DD MM YYYY — Colombian format)
- "31 01 2026" → "2026-01-31" (space-separated DD MM YYYY)

⚠️ "Doc Date" vs "Invoice Date" (Warren CAT, service invoices):
  "Doc Date: 01-30-26" = the DATE SERVICE WAS PERFORMED (not the invoice date)
  "Invoice Date: 02-20-26" = the DATE INVOICE WAS ISSUED (use THIS for invoice_date)

COLOMBIAN INVOICES (Chevron, Ecopetrol): Despite being Latin American,
Colombian Chevron invoices use MM/DD/YYYY format (not DD/MM/YYYY).
Look at context: if "Fecha Vencimiento" (due date) is 30 days after "Fecha",
the format is likely MM/DD/YYYY. When in doubt, check if month > 12.

ORACLE LAYOUT: Oracle invoices have "Invoice Date" as a SEPARATE label-value
pair in a left panel, NOT in the same row as Due Date. The top summary row
has Total Amount | Due Date | Invoice Number. Look BELOW the summary for
"Invoice Date  24-NOV-2025" as a standalone label-value.

RULE 6 — MULTI-DOCUMENT PDFs
════════════════════════════════════════════════════════════════

Many PDFs bundle multiple document types:
- INVOICE page → PRIMARY target. Extract from HERE.
- Purchase Order pages → Extract into purchase_order_reference section (if present)
- "ACKNOWLEDGEMENT" / "ORDER ACKNOWLEDGEMENT" → PO confirmation (extract into po_lines)
- "PURCHASE ORDER – ACKNOWLEDGEMENT REQUIRED" → Nabors-issued PO page.
  Has Nabors logo/header + supplier details + line items with account codes.
  ⚠️ The Nabors entity on this page is the BUYER, not the vendor — do NOT overwrite vendor_name.
  Extract into purchase_order_reference: po_number, po_date, buyer_name, po_lines
  (with part_number, quantity, unit_price, account_code, promised_date/need-by).
- "PACKING LIST" → Extract packing_slip_number + ship quantities for line item cross-refs
- "Commercial Invoice" / "Shipment No." → Shipping doc (cross-ref data, not separate invoice)
- "Proof of Delivery" → Delivery confirmation (tracking, received_by → notes)
- "THIS DOCUMENT IS NOT AN INVOICE" → skip for invoice, but extract PO data
- Terms & Conditions → IGNORE
- iProcurement Receipt Confirmation → IGNORE
- Transport Guide / Shipping Guide → IGNORE (but may confirm invoice number)
- Remisión Salida de Inventarios → IGNORE (inventory dispatch)
- Certificado de Aportes → IGNORE (social security certificate)
- Comprobante de la transacción → IGNORE (bank transfer receipt)

⚠️ DUPLICATE PAGE DETECTION:
Some PDFs contain the same page scanned twice (e.g., two copies of the invoice,
fax confirmations duplicating the original, or identical packing slips on consecutive pages).
If two pages have IDENTICAL or near-identical content:
  - Same invoice number + same line items + same totals → extract ONLY ONCE
  - Same packing list number on two pages → extract once, skip duplicate
  - Do NOT create duplicate line_items or duplicate po_lines from duplicate pages
  Clues: identical headers, identical totals, consecutive pages with same layout.

⚠️ IMPORTANT: PO total often DIFFERS from invoice total (e.g., PO = 61,409,986 COP
vs Invoice = 63,857,038 COP). ALWAYS use the INVOICE page total, never the PO total.

How to identify the invoice page: Has "INVOICE" or "Factura" as title/header,
Invoice Number, Invoice Date, Bill To, line items, and a Total.

If multiple invoices from DIFFERENT vendors exist:
- Extract from the FIRST invoice
- Note others: "Additional invoice on page [N]: [Vendor] #[Number] for [Amount]"

RULE 7 — LINE ITEMS AND AMOUNTS
════════════════════════════════════════════════════════════════

PRICE COLUMNS (NOV-style):
- "Unit Price" = list/catalog price → unit_price field
- "Net Price" / "Ext Net Price" = actual billed → amount field
- If Qty × UnitPrice ≠ Amount, a per-line discount was applied (normal)

SUBSCRIPTION/LICENSE (Thomson Reuters, software):
- "Qty/User" = informational, NOT a multiplier
- "Price" = total price for all users (already extended)
- Qty × Price ≠ Amount — expected for subscriptions

BUNDLED ITEMS (Data Foundry, colocation):
- Some items have Quantity but NO Rate and NO Amount (included with parent)
- Example: "208v 30Amp AC Power" Qty=2, no Rate/Amount → bundled with Cabinet
- Set unit_price=null, amount=null (NOT 0.00)
- Line items sum will NOT match subtotal — expected for bundled pricing

TAX FLAG (Oracle):
- "Y"/"N" in Tax column = taxable flag, NOT a unit of measure
- Set unit="" for these, NOT "Y"

CLOUD/SaaS (AWS, Azure, GCP):
- Each non-zero service IS a line item (e.g., "Amazon CloudWatch" $70.12, "Amazon RDS" $1,322.30)
- For AWS: unit_price = pre-tax Charges, tax = "Estimated US sales tax", amount = total
- quantity = 1, unit = "MO" (monthly service)
- SKIP services with $0.00 total
- SKIP linked account detail pages (page 7+) — extract from "Detail for Consolidated Bill" only
- Extract AWS Support as a line item too (e.g., "AWS Support (Business)" $100.00)

RULE 8 — PO NUMBER AND ACCOUNT NUMBER
════════════════════════════════════════════════════════════════

PO labels: "PO #", "Cust PO", "Customer PO", "P.O. Number",
"Customer Reference", "CUSTOMER ORDER NO.", "Orden de Compra"
→ po_number field

⚠️ PO PRIORITY: When a document has MULTIPLE PO-like numbers:
- "PO 40069819" (Nabors PO) and "Orden de Compra 113131001873422" (supplier's internal order)
- ALWAYS prefer the shorter Nabors PO number (usually 8 digits starting with 4)
- The supplier's "Orden de Compra" is their internal reference — put in notes if needed
- "PEDIDO" is the supplier's internal order/dispatch number — NOT the PO number
- Strip "PO"/"P0" prefix if vendor added it: "PO40069833" → "40069833"

Account labels: "Customer No.", "Acct #", "Account No", "Client No.",
"Account No:" (tax statements)
→ account_number field

⚠️ ACCOUNT NUMBER IS NOT A BANK ACCOUNT:
- "Account Number: 1335054267" in a "Wire Payments To:" or "Remit To:" section
  is a BANK account for receiving payments — do NOT put in account_number field.
- account_number is the CUSTOMER'S account with the vendor (e.g., "20406400")

⚠️ RIG NUMBER VALIDATION:
- rig_number must be an actual drilling rig identifier (e.g., "M47", "LR222", "B10", "B19")
- "NABORS LUX 2" is a Nabors LEGAL ENTITY name, NOT a rig number → leave rig_number empty
- "RIG NABORS M47 RUBIALES" → rig_number = "M47" (extract just the rig code)

Due date labels: "Due Date", "Payment Due", "Net Due", "Due By", "Pay By",
"Fecha de Vencimiento", "Fecha Vencimiento", "Vencimiento"
→ due_date field (YYYY-MM-DD)

Remit To labels: "Remit To", "Make Checks Payable To", "Please make payments to:",
"Wire Payments To:" (bank wire instructions → extract full bank details)

Notes field — capture these if present:
- Billing period: "Services for month of November", "18-Nov-2025 - 17-Feb-2026"
- Oracle Agreement: "Agreement US-CSA-CPQ-2004789"
- Oracle End User: "End User NABORS CORPORATE SERVICES INC"
- Wire/ACH payment instructions
- Special billing notes or references

SO CONFIRMATION NUMBER (so_confirmation_number):
The vendor's Sales Order or confirmation number — their acknowledgement of the PO.
Labels: "Sales Order No:", "Sales Order:", "SO Confirmation:", "Order Confirmation",
"S.O. #", "Order No:", "Confirmation Number"
- CPC: "Sales Order: 0002180-0000" → so_confirmation_number = "0002180-0000"
- Canrig modern: "Sales Order No: M150-214714" → so_confirmation_number = "M150-214714"
- Canrig older: "Order No: 123456" → so_confirmation_number = "123456"
- Cameron/SLB: "Sales Order: ..." → so_confirmation_number
This is NOT the invoice_number, NOT the po_number. It sits between PO and invoice in the
order lifecycle: PO → SO Confirmation → Shipment → Invoice.
If line items also show per-line "Sales Order" values, put those in line-level
sales_order_number AND set so_confirmation_number at header level.

PO LINE MATCHING BY PART NUMBER:
When the PDF contains BOTH invoice lines AND PO/Acknowledgement lines:
  1. After extracting both sets, compare part_number fields
  2. If invoice line part_number == po_line part_number (exact, case-insensitive):
     → Copy the PO line's line_number → invoice line's po_line_number (if not already set)
     → Copy the PO line's quantity → invoice line's po_quantity (if not already set)
  3. This enables downstream 3-way match: Invoice Line ↔ PO Line ↔ Receipt
  4. Only match on IDENTICAL part_numbers — do NOT fuzzy-match descriptions
  5. If no part_number match found, leave existing values as-is

RULE 10 — SPANISH / LATIN AMERICAN DOCUMENTS
════════════════════════════════════════════════════════════════

"Factura" = Invoice, "Factura Electrónica de Venta" = Electronic Sales Invoice
"NIT" = Tax ID (Colombia), "IVA" = VAT, "RUC" = Tax ID (Peru/Ecuador)
"Fecha" = Date, "Vencimiento" = Due Date, "Cantidad" = Quantity
"Número" = Number (invoice number), "Unidad" = Unit
"Precio Unitario" = Unit Price, "Valor" = Amount/Value
"TOTAL A PAGAR:" = Total to Pay, "Cond. De Pago" = Payment Terms
"Despachado a:" = Shipped To, "Facturado a:" = Billed To
Currency: Colombian = COP, Mexican = MXN, Argentine = ARS

COLOMBIAN INVOICES (Chevron, Ecopetrol):
- Dates often use MM/DD/YYYY (not DD/MM like other LatAm countries)
- "Número 46-1241015" → invoice_number = "46-1241015"
- "Fecha 02/03/2026" → invoice_date = "2026-02-03"
- "Fecha Vencimiento 03/05/2026" → due_date = "2026-03-05"
- "No. Orden" / "Orden de Compra" = supplier's order number (not Nabors PO)
- Look for "PO XXXXXXXX" on the invoice page for the Nabors PO number

COLOMBIAN INVOICES (Terpel):
- Dates use DD MM YYYY SPACE-SEPARATED: "03 02 2026" = Feb 3, "31 01 2026" = Jan 31
- "FACTURA ELECTRONICA DE VENTA N°" → invoice number label
- "FECHA Y HORA DE GENERACION" → invoice date (strip time portion)
- "FECHA VENCIMIENTO" → due date (also space-separated)
- "VALOR A PAGAR" → total amount
- "ORDEN DE COMPRA" → PO number (may have "PO"/"P0" prefix → strip it)
- "PEDIDO" → internal Terpel order number (NOT the PO number)
- "REMISION/BDT" → remittance/dispatch reference (put in notes)

MULTI-PAGE Colombian PDFs may also include (IGNORE all of these):
- Remisión Salida de Inventarios (inventory dispatch note)
- Certificado de Aportes (social security certificate)
- Comprobante de la transacción (bank transfer receipt)
- Purchase Order pages, T&C pages, iProcurement Receipt

- Look for Nabors PAYMENT STUB at bottom of page with:
  "INVOICE:", "INVOICE DATE:", "VENDOR:", "TOTAL PAID:", "BU:", "DESCRIPTION:"
- invoice_number: from stub "INVOICE: 2025-164211-NL2"
- invoice_date: from stub "INVOICE DATE: 2025-12-31"
- total_amount: from stub "TOTAL PAID: $87,713.00" (combined payment for all accounts)

RULE 13 — OUTPUT FORMAT
════════════════════════════════════════════════════════════════

Return ONLY valid JSON. No markdown ```json```, no preamble, no explanation.

Include a "confidence" object rating your certainty (0.0–1.0) for each critical field:
  1.0 = clearly visible label + value, no ambiguity

  0.0–0.2 = could not find it, returning empty/null

RULE 15 — DUAL TEXT EXTRACTION FLAGS
════════════════════════════════════════════════════════════════

The extracted text uses source markers per page. Interpret them as follows:

- "[SOURCE: PdfPig and Tesseract agree]"
  → Both text engines produced the same content. Use vision too to Trust the text fully.

- "[SOURCE: PdfPig only — Tesseract found no text on this page]"
  → Trust the PdfPig text. Verify against the page image if anything looks off.

- "[SOURCE: Tesseract OCR only — PdfPig found no text on this page]"
  → Likely a scanned/image-based page. Verify all critical values against the page image.

- "[SOURCE: PdfPig and Tesseract DISAGREE — verify against page image, use vision as tiebreaker]"
  → Two text blocks follow, separated by "--- PdfPig text ---" and "--- Tesseract OCR text ---".
  → READ THE PAGE IMAGE CAREFULLY. The page image is the source of truth.
  → Common disagreements:
    • PdfPig captured a barcode, watermark, or hidden text Tesseract didn't see
    • Tesseract misread characters (1↔l, 0↔O, ;↔:, S↔5)
    • Column boundaries merged differently in one of the two
    • PdfPig may have merged QTY and ITEM NO. columns on certain layouts
  → Pick the value that matches what you SEE in the page image.
  → Lower confidence to 0.7 max for any field where the two text sources disagreed.
