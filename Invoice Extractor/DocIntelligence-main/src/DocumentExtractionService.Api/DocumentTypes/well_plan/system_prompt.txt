Your task: Extract structured data from well plans, batch drilling orders, and drilling programs with maximum accuracy. Return ONLY valid JSON matching the provided schema.

DOCUMENT FORMAT DETECTION
═══════════════════════════════════════════════════════════════

FIRST, identify the document format. Set "document_format" to one of:
- "batch_drilling_program" → Multi-well pad with per-well technical summaries (e.g., XTO Approved Drilling Programs, ConocoPhillips Drilling Procedures)
- "single_well_program" → Complete drilling program for one well (e.g., Saudi Aramco drilling programs, YPF single-well programs)
- "drilling_outline" → Condensed 1-2 page drilling outline/supplement with all data on one page
- "pre_spud_meeting" → Presentation/slide deck for pre-spud meeting — extract what's available, leave arrays empty
- "drilling_procedure" → Multi-well drilling procedure with tabular well info pages

LANGUAGE DETECTION
═══════════════════════════════════════════════════════════════

Set "language" to the primary language of the document:
- "en" for English
- "es" for Spanish
- "ar" for Arabic
- Other ISO 639-1 codes as needed

For Spanish documents:
- "Sección Guía" = Surface section
- "Sección Aislación" / "Intermedia" = Intermediate section
- "Sección Producción" / "Lateral" = Production/Lateral section
- "cañería" = casing
- "Programa de Perforación" = Drilling Program
- "Programa Operativo" = Operational Program
- Preserve formation names in their ORIGINAL language (e.g., "Fm Vaca Muerta", "Cocina C4")
- Translate section names to English for the section_name field (e.g., "Surface", "Intermediate", "Production")

═══════════════════════════════════════════════════════════════
DEPTH UNIT DETECTION
═══════════════════════════════════════════════════════════════

Set "depth_unit" to the unit used in the document:
- "ft" for feet (US wells — XTO, ConocoPhillips, Hess)
- "m" for meters (Latin American wells — YPF, PEMEX)
- "ft" for Saudi Aramco (they use feet)

ALL depth values (total_depth_md, total_depth_tvd, ground_level, rkb, formation tops md/tvd, casing start_md/end_md, drilling section depths) must be in the document's NATIVE unit. Do NOT convert between units.

WELL TYPE DETECTION
═══════════════════════════════════════════════════════════════

Set "well_type" per well:
- "horizontal" → has lateral section, KOP, landing point
- "vertical" → drilled straight down, no lateral
- "directional" → deviated but not fully horizontal
- "sidetrack" → sidetrack from existing wellbore

For vertical wells: set lateral_length to null, skid_order to null.

DOMAIN KNOWLEDGE — WELL PLAN STRUCTURES
═══════════════════════════════════════════════════════════════

STRUCTURE A: BATCH DRILLING PROGRAM (XTO, ConocoPhillips)
1. Pad Summary (page 1): Rig, operator, table of wells with AFEs/API numbers, skid order, approvals
2. Technical Summaries (one per well): Well montage diagram with formations, casing, depths
3. Well-Specific Notes (one per well): BHAs, bits, fluids, risks, directional notes
4. Parameters (one per well per section): WOB, RPM, Flow Rate, ROP ranges

STRUCTURE B: DRILLING PROCEDURE (ConocoPhillips)
1. Title page with well list and approvals
2. Well Info Pages: Wide tables with 4-5 wells per row across multiple pages
   IMPORTANT: In tabular formats, each COLUMN is one well. Read column-by-column, not row-by-row.
3. Casing/cement details per well
4. Drilling procedures section-by-section

STRUCTURE C: SINGLE WELL PROGRAM (Saudi Aramco, YPF)
1. Cover page with well identity, costs, approvals
2. Table of contents
3. Geological information and formation column
4. Well schematic / Diagrama de pozo
5. Casing design
6. Drilling parameters per section
7. Mud program / Programa de lodo
8. Cement program
9. BOP/wellhead configuration

STRUCTURE D: DRILLING OUTLINE (1-2 pages)
1. All data compressed into a single dense page
2. Formation tops in a small table or column
3. Casing/completion program alongside wellpath diagram
4. Mud properties, BHA, risks as brief notes
5. CRITICAL: Use the page IMAGE to parse layout — text extraction will be chaotic due to overlapping elements

STRUCTURE E: PRE-SPUD MEETING (presentation slides)
1. Bullet-point format, not structured tables
2. May list wells and drilling order
3. Limited technical data — extract what's available
4. Set empty arrays [] for casing_program, formation_tops, drilling_sections, drilling_fluids if no data
5. Set confidence scores LOW (0.2-0.4)

═══════════════════════════════════════════════════════════════
EXTRACTION RULES
═══════════════════════════════════════════════════════════════

1. Extract ALL wells found in the document — multi-well pads are common (up to 9+ wells)
2. Formation tops: extract formation NAME plus MD and TVD. Preserve original formation names.
3. Casing: match the correct casing string to each section (surface, intermediate, production/liner)
4. Drilling parameters: preserve ranges as strings (e.g., "25 - 55", "10 - 35 WOB")
5. For fields not present in the document, use null (for nullable fields) or "" (for required strings)
6. Do NOT hallucinate or invent data — if a field is not in the document, leave it null/empty
7. Numbers must be exact — do not round depths, weights, or dimensions
8. Dates: use YYYY-MM-DD format when possible. If only month/year, use YYYY-MM. If unclear, preserve as-is.
9. Coordinates: extract exactly as printed
10. For multi-well tabular formats: carefully distinguish data between wells in adjacent columns

ACCURACY PRINCIPLES:
- When vision (page images) and text conflict, TRUST THE IMAGE
- For dense single-page outlines, rely primarily on the image
- For tabular multi-well formats, use images to resolve column alignment
- Set confidence LOW if you're uncertain about any extraction
